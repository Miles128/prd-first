# prd-first:编码前先写 PRD

vibecoding 最大的失败模式是"边问边做、边做边改",AI 写出一堆方向错误的代码。本技能的职责是在 AI 开始写业务代码前**拦一刀**,确保 `documents/prd/PRD.md` 存在且必填项齐备。这是引导式而非硬 gate——目的是帮用户想清楚,不是设路障。

## 何时触发

用户表达了"开始编码"的意图,典型措辞:
- "帮我做个 todo 应用 / 写个脚本 / 搭个 API"
- "我想实现一个 ___"
- "我们来做 ___ 项目"
- 在某个目录开始新工作,准备创建源文件

也包含显式调用 `/skill prd-first`。

## 前置检查:prd 命令是否可用

先确认 `prd` CLI 已安装。运行 `prd --help`(用 Bash 工具)。若报命令不存在,告诉用户:

> prd-first CLI 未安装。安装方式见项目 README:`cd ~/Documents/My\ Projects/prd-first && uv pip install -e .`(需先 `source .venv/bin/activate`)。
> 你也可以让我直接用问答方式口头生成 PRD,但 CLI 版本能持久化、支持断点续答,推荐安装。

不要在 CLI 未装时反复尝试;要么引导安装,要么口头走完流程并把结果写入 `documents/prd/PRD.md`。

## 核心工作流

按当前目录状态分支处理:

### 分支 1:已有 `documents/prd/PRD.md`

1. 运行 `prd check` 读取完整度。
2. 看输出:
   - **必填 100%** → 直接读取 `documents/prd/PRD.md` 全文,据其"范围""非目标""验收标准"开始编码。**严格遵守非目标清单**,用户说"顺便加个 X"时,引用 PRD 的非目标拒绝或提醒先更新 PRD。
   - **有缺失必填项** → 向用户展示缺失清单,说:
     > PRD 还有 N 项必填未确定:`<清单>`。建议先 `prd edit <key>` 补全,避免方向跑偏。要现在补吗?还是直接跳过开始编码?
     用户坚持跳过则尊重,但在编码中遇到相关决策时,提醒"这一项 PRD 里没定,我先按 ___ 假设,你看行不行"。

### 分支 2:没有 `documents/prd/PRD.md`

不要直接开始写代码。先说:

> 这个项目还没 PRD。vibecoding 前先花 5 分钟把需求想清楚,能省掉后面几小时的返工。
> 我可以引导你用 `prd init` 走一遍问答(交互式,会逐个问关键问题),或者你直接口述想法、我帮你整理成 PRD。你想怎么走?

**强烈推荐走 CLI**:`prd init`。它会:
1. 问你项目类型(web 应用 / CLI 工具 / AI agent / 后端管道)
2. 逐个问关键问题,每个问题带"为什么需要这个"的提示
3. 生成 `documents/prd/PRD.md` + `documents/prd/meta.yaml`,支持中途退出、断点续答

如果用户选择你口述整理,按下方"四类项目模板"的必填字段逐个问用户,问完写入 `documents/prd/PRD.md`。

### 分支 3:用户就是不想写 PRD

尊重用户决定,但**明确告知代价**:
> 好,跳过 PRD 直接开始。提醒一下:遇到模糊的需求我会做假设并告诉你,如果方向不对随时打断。建议编码中一旦发现"我又在猜用户的意图了",就停下来补 PRD。

然后正常编码,但在每个有歧义的决策点显式说出你的假设。

## 编码中的持续约束

PRD 写完后,编码期间持续遵守:
- **非目标清单**是硬约束。用户说"顺便加个登录吧",而 PRD 非目标写了"不做认证",你要提醒:"PRD 里写的不做认证,要先更新 PRD 吗?"而不是默默加上。
- **验收标准**是完成判断。任务做完后,对照 PRD 的验收标准逐条核对,而不是凭感觉说"差不多了"。
- **技术栈约束**要遵守。PRD 写了用 Python 就别用 Node,除非先更新 PRD 说明理由。

## 四类项目模板(用于无 CLI 时口头引导)

问问题时,每个问题都先说"为什么问这个"(帮用户理解意图),再问。

### Web 应用(web-app)
必填:问题陈述 / 目标用户 / 成功定义 / MVP 范围 / 非目标 / 核心场景 / 页面结构 / 技术栈约束 / 验收标准
可选:用户流 / 认证方式 / 数据模型 / API 端点 / 状态管理 / 部署

### CLI / 脚本工具(cli-tool)
必填:问题陈述 / 目标用户 / 成功定义 / 命令结构 / MVP 范围 / 非目标 / 输入格式 / 输出格式 / 依赖项 / 验收标准
可选:退出码 / 安装方式

### AI / Agent 项目(ai-agent)
必填:问题陈述 / 目标用户 / 成功定义 / Agent 角色 / 输入契约 / 输出契约 / 失败模式与降级 / 评估方式 / 非目标 / 验收标准
可选:工具清单 / 成本延迟约束 / 安全护栏
**重点**:AI 项目必须逼出"评估方式"和"量化成功定义",否则 prompt 调不准。

### 后端 / 数据管道(backend-data)
必填:问题陈述 / 成功定义 / 数据来源 / 数据流图 / 外部依赖 / 容错策略 / 监控告警 / 非目标 / 验收标准
可选:存储方案 / schema / 触发方式 / 幂等设计
**重点**:必须问"出错怎么办"和"监控哪些指标",后端项目 90% 代码在处理失败。

## 引导式而非强制的分寸

- CLI 从不拒绝生成 PRD,缺字段也能产出(留 `_(待补充)_` 占位)。
- 你也不拒绝编码,只是**先确认 PRD 状态**,缺失时**提醒而非阻拦**。
- "强制力"来自:问答本身的引导性(每个问题带 why)+ 你在编码中持续回引 PRD 缺口。
- 用户说"skip prd"或明确拒绝,就不再追问,进入分支 3。

## 不要做的事

- 不要在 PRD 缺失时默默开始写一堆代码——至少要先说一句话提醒。
- 不要写完 PRD 就忘掉它——编码中持续回引范围/非目标/验收标准。
- 不要把 PRD 当成一次性文档——需求变了就 `prd edit` 更新,保持 PRD 与代码同步。
- 不要用硬 gate 态度("PRD 没写完不准写代码!")——这是工具,不是审批流程。
