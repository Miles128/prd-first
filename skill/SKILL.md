# prd-first: 编码前先写 PRD

vibecoding 最大的失败模式是"边问边做、边做边改"。本技能的职责是在 AI 开始写业务代码前**拦一刀**，
但拦住的方式不是死板表单，而是**grill-me 式的主动追问**：把项目模板当作初始决策树地图，
沿每个分支深挖依赖、边界和失败模式，直到达成共识，再把结论写入 `documents/prd/PRD.md`。

## 何时触发

用户表达了"开始编码"的意图，例如：
- "帮我做个 todo 应用 / 写个脚本 / 搭个 API"
- "我想实现一个 ___"
- "我们来做 ___ 项目"
- 在某个目录开始新工作，准备创建源文件

也包含显式调用 `/skill prd-first`。

## 前置检查

1. 先确认 `prd` CLI 是否已安装：运行 `prd --help`。
2. 若未安装，告诉用户：
   > prd-first CLI 未安装。推荐安装：`cd ~/Documents/My\ Projects/prd-first && uv pip install -e .`（需先激活 .venv）。
   > 也可以让我直接口头走完流程并把结果写入 `documents/prd/PRD.md`，但 CLI 支持持久化和断点续答。
3. 检查当前目录是否已有 `documents/prd/PRD.md`：
   - 有 → 用 `prd check` 看完整度，完整则按 PRD 编码；不完整则提醒补全。
   - 没有 → 进入下方的 drill + PRD 流程。

## 核心工作流：drill-first，PRD-second

### 1. 加载决策树地图

根据项目类型读取对应参考文件：
- `src/prd_first/assets/drill-guides/<type>.yaml`（决策树追问指南）
- `src/prd_first/assets/templates/<type>.yaml`（模板字段及每个字段的 `drill_questions`）

**模板和 drill-guides 只是参考，不是强制表单**。你需要用它们构建初始决策树地图，
但应根据用户的实际回答动态增删分支。

如果没有明确项目类型，先问用户："这是 web 应用、CLI 工具、AI agent 还是后端/数据管道？"

### 2. 展示地图并征得同意

用一句话向用户展示你打算追问哪些分支，例如：
> 我想从这几个分支把你的计划 drill 清楚：问题陈述、目标用户、成功定义、MVP 范围、非目标、页面结构、技术栈、验收标准。过程中你可以随时添加新分支。可以吗？

### 3. 逐分支 relentless interview

对每个分支，遵守 grill-me 的核心规则：

- **一次只问一个问题**。不要一次性抛出 3 个以上问题。
- **每个问题都给出你的推荐答案**，让用户可以"同意"或"修正"，而不是从零发明。
- **沿着决策树往下走**：先问基础分支（问题陈述、目标用户），再问依赖它的分支（范围、技术栈、验收标准）。
- **深挖隐藏假设**：当用户回答模糊或你发现有未声明的依赖时，继续追问，直到双方对答案、边界和依赖达成共识。
- **能查代码库就查代码库**：如果问题可以通过看现有代码回答，先用代码库工具探索，再问用户确认。
- **允许添加计划外分支**：如果用户提到成本、安全、性能、合规等重要话题，把它们加入决策树并追问清楚。

每个分支结束时问：
> 关于「分支名」我们是否已经达成共识？如果还有没想清楚的就继续问，清楚了就进入下一分支。

### 4. 收敛到 PRD

所有分支结束后：
1. 用 `prd init <type>` 或纯问答把结论填入 PRD 模板字段。
2. 生成/更新 `documents/prd/PRD.md` 和 `documents/prd/meta.yaml`。
3. 运行 `prd check` 确认必填完整度。
4. 向用户展示生成的 PRD 摘要，特别是**范围、非目标、验收标准**。

## 追问示例（以 AI agent 为例）

**用户**：我想做一个自动回复客服邮件的 AI agent。

**AI**：
> 先确认一下目标用户：是客服团队自己用，还是最终客户直接收到 AI 回复的邮件？
> 我的推荐答案：如果是客服团队用，AI 生成草稿后由人工审核再发送；如果是客户直接收到，需要更严格的安全护栏和免责声明。你倾向哪种？

**用户**：客户直接收到，但只处理退款查询。

**AI**：
> 明白了，范围限定在退款查询。那失败模式分支：如果 AI 误判了退款资格并自动回复了错误承诺，最坏后果是什么？是否需要人工审核回路？
> 我的推荐答案：退款涉及金额，应该有置信度阈值，低于阈值时转人工而不是直接回复。

## 编码中的持续约束

PRD 生成后，编码期间持续遵守：
- **非目标清单是硬约束**：用户说"顺便加个 X"，如果 X 在非目标里，提醒先更新 PRD。
- **验收标准是完成判断**：任务做完后对照 PRD 验收标准逐条核对。
- **技术栈约束要遵守**：PRD 写了 Python 就别用 Node，除非先更新 PRD。

## 不要做的事

- 不要直接把模板字段当成填空题一次性问完。
- 不要用户一回答就停止追问关键分支，要问到共享理解。
- 不要拒绝用户偏离模板——模板是参考，决策树可以生长。
- 不要写完 PRD 就忘掉它——编码中持续回引范围/非目标/验收标准。
- 不要用硬 gate 态度（"PRD 没写完不准写代码!"）——这是工具，不是审批流程。
